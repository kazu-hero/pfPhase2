# 18 Error Stop Protocol

## 目的
- エラー発生時に自動的に停止し、推測による別案を防ぐ。

## エラー発生時の絶対ルール

```
【エラー発生】
   ↓
【即座に停止】
   ↓
【停止報告を提示】
   ↓
ユーザーの指示をお待ちします
   ↓
（ユーザーの新指示があるまで動かない）
```

## 停止報告のフォーマット（固定）

エラー発生時は、以下の形式で必ず停止報告を提示：

```
【エラー停止】

概要:
[エラーの要約]

参照定義:
[02_solution_definition.md のどの項目に関連するか]

影響範囲:
[どの機能/ファイルに影響するか]

発生条件:
[どの操作時に発生したのか]

エラーメッセージ:
[正確なメッセージ]

仮説（以下から選択）:
1. [原因の仮説1]
2. [原因の仮説2]
3. [原因の仮説3]

次の確認項目:
1. [確認すべきこと1]
2. [確認すべきこと2]
3. [確認すべきこと3]

ユーザーの判断をお待ちします。
```

## 禁止事項（絶対に違反するな）

✗ **エラー発生後に勝手に別案を試す**
（例：設定参照が失敗 → 仕様外の代替設定を作成、など）

✗ **推測で対応する**
（例：「だからこういう方法にしましょう」と勝手に設計変更）

✗ **スコープ外の「改善案」を導入する**
（例：セキュリティ強化を理由に仕様外の設定を追加、など）

✗ **エラーを無視して先に進む**

✗ **ユーザーに報告していないまま次のステップに進む**

## 許可されている対応（限定）

以下のみエラー対応として自動実行OK：

### 1. コンパイル時の「明らかなtypo修正」のみ
- 条件：
  - シンタックスエラーで「Unexpected token」等の明確なtypo
  - 修正が1行のみ
  - 02_solution_definition.md に矛盾していない
- 対応：typoを修正 → 修正内容を報告
- 例：`String = "value"` → `String x = "value";`

### その他のエラー（全て停止）
- 型の不一致、ロジックエラー、デプロイ順序問題、ファイル存在確認エラー
  **→ 全て 18_error_stop_protocol に従い停止報告**
- 理由：修正の妥当性判定にユーザーの指示が必須

## 例：テンプレートとしての失敗例と正しい対応

### ❌ 失敗例（禁止）
```
【エラー】設定参照式 [参照式A] が無効です

【違反】勝手に別案を試す
→ 仕様にない代替コンポーネントを新規作成
→ スコープ外なのに無視して展開
```

### ✅ 正しい対応（今後）
```
【エラー停止】

概要:
設定参照式 [参照式A] が解決されない

参照定義:
02_solution_definition.md の該当仕様セクション
関連する公式ドキュメント

影響範囲:
[機能A] の設定参照が機能しない

仮説:
1. 参照キー/パラメータ名が不一致の可能性
2. 現行バージョンで参照式が未サポートの可能性
3. 関連設定値のいずれかが無効

次の確認項目:
1. 実環境の設定キー名を確認
2. 関連設定の有効化状態を再確認
3. 公式ドキュメントで正しい参照構文を確認

ユーザーの判断をお待ちします。
```

この場合は、一切別案を試さずに待つ。
