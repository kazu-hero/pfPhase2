# 17 Approval Before Deploy

## 目的
- コード修正後、展開前に必ずユーザーの承認を得る強制フロー。

## 実装後～展開前に必ず実行

### ステップ1：修正内容を整理して提示

以下の形式で**必ず差分を明示**して、ユーザーの確認を待つ：

```
【修正内容のお知らせ】

修正ファイル: [ファイルパス]

修正前（既存コード）:
[修正前のコード10行程度]

修正後（新規コード）:
[修正後のコード10行程度]

変更理由:
[なぜ必要か、どう改善されるか]

新規作成: [はい/いいえ]
スコープ内確認: [はい/いいえ]
```

### ステップ2：複数ファイル修正の場合

各ファイルについて上記フォーマットを繰り返す。

```
【修正1】: [ファイル1]
[差分]

【修正2】: [ファイル2]
[差分]

...

【全体確認】
以上の修正で間違いないでしょうか？
展開してもいいですか？
→ ユーザーの「OK」確認まで待機
```

### ステップ3：「OK」確認後に展開

ユーザーから明示的な「承認」を得たら、以下を実行：
1. 対象検証環境に展開
2. change_log.md に変更を記録
3. 変更結果を報告

### ステップ4：デプロイ後の変更ログ記録

change_log.md に以下の形式で追記：

```markdown
### [日時] [作業内容]
- ファイル: [パス]
- 変更内容: [簡潔な説明]
- ユーザー指示: [何のために]
- デプロイ結果: [成功/失敗]
```

## 禁止事項

✗ **ユーザーの明示的な「OK」なしに展開する**
✗ **複数修正をまとめて提示する（1つずつ確認）**
✗ **修正前後のコードを提示せずに「修正しました」と報告する**
✗ **デプロイ結果を報告しない**

## 例外ケース

**原則として例外ケースは存在しません。**

ただし、以下の場合のみ自動修正が許可されます（18_error_stop_protocol参照）：

### 許可される自動修正
- **コンパイル時の明らかなtypo修正のみ**
  - 条件：シンタックスエラーで「Unexpected token」等の明確なtypo
  - 条件：修正が1行のみ
  - 条件：02_solution_definition.md に矛盾していない
  - 対応：typoを修正 → 修正内容を報告
  - 例：`String = "value"` → `String x = "value";`

### その他の全ての修正
全ての修正（バグ修正、ロジック修正、複数行の修正等）は、展開前にユーザーの承認が必須です。

「これはバグ修正だから」という AIの判断で例外扱いすることは禁止。
