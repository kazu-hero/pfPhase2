## 1. ユースケース概要

### 1.1 概要
ユーザーが相談ベースで要望をチャット入力し、Agentが要望を分析・構造化して、以下の一連のフローを実行：
1. `Dev_Service__c`（ユーザサービス）を作成
2. 抽出した課題・要望を`Issue__c`に起票
3. `Relation_Support2Issue__c`でリレーションを作成

### 1.2 アクター
- **Primary**: 社内ユーザー（開発知識なし、ITリテラシー: Excel操作程度）
- **Secondary**: Agentforce Agent

### 1.3 前提条件
- ユーザーがAgentforceチャットにアクセス可能
- ユーザーが`Dev_Service__c`、`Issue__c`の作成権限を持つ

### 1.4 事後条件（成功時）
- `Dev_Service__c`レコードが作成されている
- 1つ以上の`Issue__c`レコードが作成されている
- `Relation_Support2Issue__c`でリレーションが作成されている
- 全レコードのステータスが適切に設定されている

## 2. フロー全体図

```
[ユーザー: 要望入力]
    ↓
[Agent: 要望理解・確認]
    ↓
[Agent: 要望構造化]
    ↓
[Agent: Dev_Service__c作成確認] ← P-001パターン
    ↓（承認）
[Agent: Dev_Service__c作成] ← createRecord Action
    ↓
[Agent: 課題抽出提示]
    ↓
[ユーザー: 課題確認・修正]
    ↓
[Agent: Issue__c作成確認（複数）] ← P-001パターン
    ↓（承認）
[Agent: Issue__c作成（ループ）] ← createRecord Action
    ↓
[Agent: Relation作成] ← createRelation Action
    ↓
[Agent: 完了報告] ← P-006パターン
    ↓
[Agent: 次アクション提案] ← P-007パターン
```

## 3. 詳細フロー設計

### 3.1 Step 1: 要望入力受付

#### ユーザー入力例
```
「月次売上レポートに顧客別の内訳を追加したいです。
現在は全体の売上しか見えないため、顧客ごとの貢献度が把握できません。
また、レポートを自動でメール送信できるようにしたいです。」
```

#### Agent応答（要望理解・確認）
```
ご要望を承りました。以下の内容で間違いないでしょうか？

【要望内容】
1. 月次売上レポートに顧客別の内訳を追加
   - 現状: 全体の売上のみ表示
   - 課題: 顧客ごとの貢献度が把握できない

2. レポートの自動メール送信
   - 現状: 手動でレポートを作成・送信
   - 課題: 作業の手間がかかる

この理解で正しいでしょうか？
修正や追加があれば教えてください。
```

#### Agent応答（作成確認）
```

- ステータス: OPEN（対応中）
- 内容: 月次売上レポートに顧客別の内訳を追加し、自動メール送信機能を実装したい。

この内容でレコードを作成してよろしいですか？
「はい」「実行」「OK」などで承認いただければ処理を開始します。
```

#### Apex処理（createRecord Action）
```apex
Map<String, Object> fieldValues = new Map<String, Object>{
```

### 3.5 Step 5: Issue__c作成

#### Agent応答（作成確認）
```
【確認】課題・要望レコード作成

以下の2件のレコードを作成します：

■ レコード1: 顧客別内訳の追加
- オブジェクト: 課題・要望（Issue__c）
- ステータス: TODO（未着手）
- カテゴリ: レポート改善
- 内容: （上記の詳細内容）

■ レコード2: レポート自動メール送信
- オブジェクト: 課題・要望（Issue__c）
- ステータス: TODO（未着手）
- カテゴリ: 業務効率化
- 内容: （上記の詳細内容）

この内容でレコードを作成してよろしいですか？
「はい」「実行」「OK」などで承認いただければ処理を開始します。
```

#### Apex処理（ループでcreateRecord Action）
```apex
List<Id> issueIds = new List<Id>();

for (Map<String, Object> issueData : extractedIssues) {
    Map<String, Object> fieldValues = new Map<String, Object>{
        'Name' => (String)issueData.get('name'),
        'Status__c' => 'TODO',
        'category__c' => (String)issueData.get('category'),
        'content__c' => (String)issueData.get('content'),
        'OwnerId' => UserInfo.getUserId()
    };
    
    Id issueId = AgentRecordController.createRecord(
        'Issue__c',
        fieldValues
    );
    
    issueIds.add(issueId);
}
```

### 3.6 Step 6: Relation作成

#### Apex処理（createRelation Action）
```apex
for (Id issueId : issueIds) {
    Map<String, Object> relationFields = new Map<String, Object>{
        'Dev_Service__c' => serviceId,
        'Issue__c' => issueId
    };
    
    AgentRelationController.createRelation(
        'Relation_Support2Issue__c',
        relationFields
    );
}
```

## 5. エラーハンドリング

### 5.1 想定エラーシナリオ

#### シナリオ1: Dev_Service__c作成失敗
- **原因**: 必須項目未入力、権限不足
- **処理**: E-001またはE-004エラー、P-003パターン適用
- **復旧**: ユーザーに再入力または管理者連絡促す

#### シナリオ2: Issue__c作成失敗（1件目は成功、2件目で失敗）
- **原因**: 権限不足、Governor Limits
- **処理**: ロールバック（Phase 3で検討）、エラー通知
- **復旧**: 成功した分を報告、失敗した分を再試行

#### シナリオ3: Relation作成失敗
- **原因**: 親レコード削除済み、権限不足
- **処理**: E-003またはE-004エラー、P-003パターン適用
- **復旧**: 親レコードの確認、権限の確認

### 6.1 応答時間目標
| ステップ | 目標時間 |
|---------|---------|
| 要望理解・確認 | 3秒以内 |
| Dev_Service__c作成 | 2秒以内 |
| Issue__c作成（2件） | 3秒以内 |

| **合計** | **10秒以内** |

### 6.2 最適化戦略
- レコード作成をバルク化（Phase 3実装時）
- リレーション作成をバルク化
- SOQLクエリの最小化

## 7. バリエーション設計

### 7.1 バリエーション1: 単一課題の場合
ユーザー入力が1つのシンプルな要望の場合：
- Issue__cを1件のみ作成
- Relationも1件のみ作成
- フロー簡略化

### 7.2 バリエーション2: 複雑な要望の場合
ユーザー入力が複数の要素を含む場合：
- 課題を3件以上抽出
- 優先度付け
- 段階的な確認プロセス

### 7.3 バリエーション3: スコープ外要望を含む場合
ユーザー入力にApex実装等のスコープ外要望が含まれる場合：
- スコープ外要望を別途通知（P-005パターン）
- スコープ内要望のみをIssue__cに起票
- 代替案を提示
