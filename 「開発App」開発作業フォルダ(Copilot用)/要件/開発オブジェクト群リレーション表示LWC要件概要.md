# 開発オブジェクト群のパス表示

## **目的**

開発プロセス全体の依存関係をグラフで可視化し、ユーザサービスから開発チケットまでのパス（関連性）を一目で把握できるようにする。5種類の主要オブジェクトと4つの中間オブジェクトで構成される複雑なネットワーク構造を実装する。

## **1\. オブジェクトと関係性の整理**

### **開発オブジェクト群の全体構造**

以下の9種類のオブジェクトで開発プロセスを表現します：

**主要オブジェクト（ノードとして表示）**：
1. **Dev_Service__c（ユーザサービス）** - 開発の起点となるユーザー向けサービス
2. **Issue__c（課題・要望）** - サービスに関連する課題や要望
3. **TechActionPlan__c（対応施策）** - 課題を解決するための技術的施策
4. **FunctionRequirements__c（機能要件）** - 施策を実現するための機能要件
5. **Dev_Ticket__c（開発チケット）** - 実際の開発タスク

**中間オブジェクト（接続を管理、UI上は非表示）**：
1. **Relation_Support2Issue__c** - ユーザサービスと課題・要望の多対多関係
2. **Relation_Issue2Plan__c** - 課題・要望と対応施策の多対多関係
3. **Relation_Plan2Requirements__c** - 対応施策と機能要件の多対多関係
4. **Relation_Requirements2Dev__c** - 機能要件と開発チケットの多対多関係

### **データフロー**

```
Dev_Service__c (ユーザサービス)
    ↓ (Relation_Support2Issue__c)
Issue__c (課題・要望)
    ↓ (Relation_Issue2Plan__c)
TechActionPlan__c (対応施策)
    ↓ (Relation_Plan2Requirements__c)
FunctionRequirements__c (機能要件)
    ↓ (Relation_Requirements2Dev__c)
Dev_Ticket__c (開発チケット)
```

### **必要なデータ項目**

**主要オブジェクト共通項目**：
* ID（Salesforce標準）
* Name（名前）
* Status__c（ステータス：todo / wip / done）
* Owner（担当者）
* Due_Date__c（期限）
* Description__c（説明）

**中間オブジェクト共通項目**：
* ID（Salesforce標準）
* 上流オブジェクトID（主従関係または参照関係）
* 下流オブジェクトID（主従関係または参照関係）

### **関係の可視化のポイント**

* ユーザサービスから開発チケットまでの全ての「パス」をグラフで表示する。

* 中間オブジェクトはデータの接続を保持するが、**UI上には表示しない**（エッジとして表現）。

* ステータス（todo/wip/done）によってノードの色を変え、進捗状況を視覚的に表現する。

* 階層構造（5階層）を持つネットワークで、各レイヤー間の関係を明確に表示する。

* 分岐（1つの課題から複数の施策）や合流（複数の要件から1つの開発チケット）を表現できる設計とする。

## **2\. データの取得と処理**

### **Apexの役割**

* **データ取得**：Apexを使って、5階層のオブジェクト間の関係を取得する。

  * **階層的なクエリ**：各中間オブジェクトを経由して、関連する全てのオブジェクトを取得。
    1. Dev_Service__c から Relation_Support2Issue__c 経由で Issue__c を取得
    2. Issue__c から Relation_Issue2Plan__c 経由で TechActionPlan__c を取得
    3. TechActionPlan__c から Relation_Plan2Requirements__c 経由で FunctionRequirements__c を取得
    4. FunctionRequirements__c から Relation_Requirements2Dev__c 経由で Dev_Ticket__c を取得

  * **SOQL最適化**：必要なデータを効率的に取得するため、サブクエリや結合を活用。

  * **データ構造**：ノード（5種類の主要オブジェクト）とエッジ（4種類の関係）という形式で処理する。

  * **メタデータ取得**：各オブジェクトのステータス、担当者、期限などの情報も合わせて取得する。

* **フィルター対応のデータ取得**：

  * **起点レコードフィルター実装**：
    - オブジェクトタイプに応じて5パターンの検索ロジックを実装
    - 再帰的に上流・下流のレコードを取得
    - 訪問済みレコードをSetで管理し、無限ループを防止
    - 最大深度を10階層に制限

  * **パターン別の取得メソッド**：
    ```apex
    // パターン1: Dev_Service起点 - 下流のみ取得
    fetchDownstreamFromService(Id serviceId, GraphData result)
    
    // パターン2: Issue起点 - 上流と下流を取得
    fetchUpstreamToService(Id issueId, GraphData result)
    fetchDownstreamFromIssue(Id issueId, GraphData result)
    
    // パターン3: TechActionPlan起点 - 上流と下流を取得
    fetchUpstreamToIssue(Id planId, GraphData result)
    fetchDownstreamFromPlan(Id planId, GraphData result)
    
    // パターン4: FunctionRequirements起点 - 上流と下流を取得
    fetchUpstreamToPlan(Id reqId, GraphData result)
    fetchDownstreamFromRequirements(Id reqId, GraphData result)
    
    // パターン5: Dev_Ticket起点 - 上流のみ取得
    fetchUpstreamToRequirements(Id ticketId, GraphData result)
    ```

  * **一括クエリでN+1問題を回避**：
    ```apex
    // 悪い例：ループ内でクエリ
    for (Relation_Issue2Plan__c rel : relations) {
        TechActionPlan__c plan = [SELECT Id FROM TechActionPlan__c WHERE Id = :rel.TechActionPlan__c];
    }
    
    // 良い例：一括取得
    Set<Id> planIds = new Set<Id>();
    for (Relation_Issue2Plan__c rel : relations) {
        planIds.add(rel.TechActionPlan__c);
    }
    Map<Id, TechActionPlan__c> plans = new Map<Id, TechActionPlan__c>(
        [SELECT Id, Name, Status__c FROM TechActionPlan__c WHERE Id IN :planIds]
    );
    ```

* **中間オブジェクトの処理**：

  * 中間オブジェクト（Relation_*）自体は画面に表示しないが、エッジ（線）として接続情報を使用する。

  * 多対多の関係を正確に表現するため、全ての中間レコードを取得・処理する。

* **データの整形**：

  * 取得したデータを**JSON形式**で整理し、Cytoscape.js が理解できる形式（nodes配列とedges配列）に変換する。
  
  * ノードタイプ（Dev_Service / Issue / TechActionPlan / FunctionRequirements / Dev_Ticket）を識別子として付与。

* **詳細パネル用の追加データ取得**：

  * **選択ノードの全関連レコード取得**：
    - ユーザーがノードをクリックした際、そのレコードの全関連レコードを取得
    - **フィルター条件を無視**して、上流・下流の全レコードを取得
    - 各関連レコードに以下の情報を含める：
      * ID、Name、Status、Owner、Due_Date
      * オブジェクトタイプ
      * レコードURL生成用の情報
  
  * **フィルター状態の付与**：
    - 各関連レコードが現在のフィルター条件に合致するかを判定
    - `isFilteredOut: true/false` フラグを付与
    - フロントエンドで視覚的に区別するために使用
  
  * **効率的なデータ取得**：
    ```apex
    // 選択ノードの関連レコード取得メソッド（フィルター無視版）
    @AuraEnabled
    public static RelatedRecordsData getRelatedRecords(
        Id recordId,
        String objectType
    ) {
        RelatedRecordsData result = new RelatedRecordsData();
        
        // 上流・下流の全関連レコードを取得（フィルター無視）
        result.upstreamRecords = fetchAllUpstream(recordId, objectType);
        result.downstreamRecords = fetchAllDownstream(recordId, objectType);
        
        return result;
    }
    ```

## **3\. フロントエンド（LWC）の設計**

### **LWCの役割**

* **データ表示**：Apexから渡されたデータを基に、5階層のオブジェクト関係をグラフとして表示する。

* **グラフ表示**：**Cytoscape.js**を使って、複雑な階層構造を可視化する。
  * 軽量で高速なグラフ専用ライブラリ
  * 階層型レイアウト（Breadthfirst、Dagre）で5階層を明確に表現
  * ズーム、パン、ノードドラッグなどのインタラクション機能
  * モバイルデバイスでもスムーズに動作

* **ノードのビジュアル設計**：
  * **形状でオブジェクトタイプを区別**：
    - Dev_Service__c：六角形（サービスの起点）
    - Issue__c：円形（課題）
    - TechActionPlan__c：四角形（施策）
    - FunctionRequirements__c：菱形（要件）
    - Dev_Ticket__c：楕円形（チケット）
  
  * **色でステータスを表現**：
    - todo：グレー (#999999)
    - wip：黄色 (#FFC107)
    - done：緑 (#4CAF50)

* **エッジのビジュアル設計**：
  * 矢印で方向性を表現（上流→下流）
  * 関係タイプごとに色を変える（オプション）

* **レイアウトアルゴリズム**：
  * **Breadthfirst**または**Dagre**レイアウトを使用し、階層を明確に表示
  * 左から右、または上から下へのフローを表現

* **動的な表示**：

  * ユーザーがノードをクリックした際に、**詳細情報**を表示する仕組みを組み込む（名前、ステータス、担当者、期限、説明など）。

  * **パスハイライト機能**：特定のノードを選択すると、そのノードに関連する上流・下流のパス全体を強調表示。

  * **フィルタリング機能**：ステータスやオブジェクトタイプでフィルタリングできるようにする。

* **詳細パネルの実装仕様**：

  * **ハイパーリンク実装**：
    - `lightning-formatted-url` コンポーネントまたは `<a>` タグを使用
    - レコードIDから動的にURLを生成：`/lightning/r/{ObjectApiName}/{RecordId}/view`
    - `target="_blank"` で新しいタブで開くオプション提供

  * **フィルター外レコードの取得と表示**：
    - Apex側で、選択ノードの全関連レコードを取得（フィルター条件を無視）
    - フロントエンド側で、現在のフィルター条件と照合
    - フィルター外レコードには特別なフラグを付与
    - CSSクラスで視覚的に区別（例：`filtered-out-record`）
    - アイコンやバッジで明示

  * **関連レコードのインタラクション**：
    - 各関連レコードにホバーでツールチップ表示
    - クリック時の動作を選択可能：
      * デフォルト：レコードページへ遷移
      * オプション：詳細パネル内で情報切替（SPA的な動作）

### **グラフの表示方法**

* **ノード**：5種類の主要オブジェクトをノードとして表示。形状と色でオブジェクトタイプとステータスを識別。

* **エッジ**：ノードを繋ぐ矢印（エッジ）は、**4種類の中間オブジェクト**を介して接続される関係を示すが、中間オブジェクト自体はノードとして表示しない。

* **階層表示**：5階層の構造が視覚的に理解しやすいレイアウトで配置。

* **インタラクション**：ノードや線をクリック、ホバーすることで詳細情報が見えるようにする。

## **4\. 画面構成**

### **App Pageの設定**

* \*\*App Page（Lightning App Page）\*\*を使って、**独立したタブ**として開発プロセス全体のグラフ表示を行います。

  * App Pageは、Salesforce内で「カスタムタブ」を作成する際に使用されるページタイプで、**LWCコンポーネント**を埋め込むことができます。

  * このページには、開発オブジェクト群の関係性グラフを表示するためのLWCコンポーネントが埋め込まれ、ユーザーがこのタブを開くことでユーザサービスから開発チケットまでの全体像が視覚化されます。

### **画面構成例**

```
┌─────────────────────────────────────────────────────────────────┐
│ 📊 開発プロセス可視化                                            │
├────────────────┬────────────────────────────────────┬───────────┤
│ フィルター     │ グラフ表示領域                      │ 詳細情報  │
│                │                                    │           │
│ ▼ ステータス   │  ┌──┐                             │ [選択中]  │
│ ☑ todo         │  │SV│──→[IS]──→[PL]──→[RQ]──→[TK]│ Name:     │
│ ☑ wip          │  └──┘                             │ Status:   │
│ ☑ done         │                                    │ Owner:    │
│                │  Legend:                           │ Due:      │
│ ▼ オブジェクト │  ■ Dev_Service  ● Issue           │           │
│ ☑ Service      │  ■ TechActionPlan ◆ Requirements  │ 関連:     │
│ ☑ Issue        │  ○ Dev_Ticket                      │ ↑上流     │
│ ☑ Plan         │                                    │ ↓下流     │

│ ▼ 起点レコード │                                    │           │
│ ( ) Service    │                                    │           │
│ ( ) Issue      │                                    │           │
│ ( ) Plan       │                                    │           │
│ ( ) Req        │                                    │           │
│ ( ) Ticket     │                                    │           │
│                │                                    │           │
│ [検索...]      │                                    │           │
│                │                                    │           │
│ ☐ 上流表示     │                                    │           │
│ ☐ 下流表示     │                                    │           │
│                │                                    │           │
│ [適用] [クリア] │                                    │           │
└────────────────┴────────────────────────────────────┴───────────┘
```

* **左側フィルターパネル**：
  * 折りたたみ可能な3つのフィルターセクション
  * ステータスフィルター（チェックボックス）
  * オブジェクトタイプフィルター（チェックボックス）
  * 起点レコードフィルター（ラジオボタン + 検索 + オプション）

* **中央グラフ表示領域**：
  * LWCコンポーネントで作成した5階層グラフの表示領域
  * ズーム・パンコントロール
  * 凡例（オブジェクトタイプと形状・色の説明）
  * グラフ操作ツールバー（ズームイン/アウト、リセット、フィット）

* **右側詳細パネル**：
  * ノード（レコード）をクリックした際に表示される**詳細情報**
  * **表示内容**：
    - **Name**（レコード名）：**ハイパーリンク形式**でレコード詳細ページへ遷移可能
    - Status（ステータス）
    - Owner（担当者）
    - Due Date（期限）
    - Description（説明）
  * **関連レコード一覧**：
    - **上流レコード**：選択ノードより上位階層の関連レコード
    - **下流レコード**：選択ノードより下位階層の関連レコード
    - 各関連レコードもハイパーリンク形式で表示
    - **重要：フィルターで非表示になっている関連レコードも表示**
      * 例：ステータスが「done」でフィルター除外されていても、選択ノードと関連があれば一覧に表示
      * グラフ上では非表示だが、詳細パネルでは確認可能
      * アイコンや色で「フィルター外」であることを視覚的に示す
    - 関連レコードをクリックすると：
      * オプション1：そのレコードの詳細ページに遷移
      * オプション2：グラフ上でそのノードを強調表示（フィルター内の場合）
      * オプション3：詳細パネルの内容を切り替えて、そのレコードの情報を表示

## **5\. フィルター機能の詳細仕様**

### **5.1 ステータスフィルター（必須機能）**

* **目的**：進捗状況に応じてノードを絞り込み、作業中や完了したタスクを可視化する

* **UI設計**：
  * チェックボックス形式で複数選択可能
  * デフォルト：全て選択状態（全データ表示）

* **フィルター項目**：
  ```
  ☑ todo   （未着手）
  ☑ wip    （作業中）
  ☑ done   （完了）
  ```

* **動作**：
  * チェックを外したステータスのノードは非表示
  * 例：「done」のチェックを外す → 完了済みタスクを除外し、進行中の作業に集中
  * 例：「todo」と「wip」のみ選択 → 未完了タスクのみ表示

* **適用範囲**：
  * 全階層（全5種類のオブジェクト）に適用
  * フィルター後、孤立したノード（他のノードと接続がない）は自動的に非表示

### **5.2 オブジェクトタイプフィルター**

* **目的**：特定の階層のみを表示し、関心のある範囲に集中する

* **UI設計**：
  * チェックボックス形式で複数選択可能
  * 最低1つは選択必須

* **フィルター項目**：
  ```
  ☑ Dev_Service__c（ユーザサービス）
  ☑ Issue__c（課題・要望）
  ☑ TechActionPlan__c（対応施策）
  ☑ FunctionRequirements__c（機能要件）
  ☑ Dev_Ticket__c（開発チケット）
  ```

* **動作**：
  * チェックを外したオブジェクトタイプのノードは非表示
  * 非表示階層をスキップして、表示中のノード間を直接エッジで接続
  * 例：Issue__c と TechActionPlan__c のみ選択 → 課題と施策の関係のみを可視化

### **5.3 起点レコードフィルター（重要機能）**

* **目的**：特定のレコードを起点に、そこから派生する上流・下流の関連レコードのみを表示する

* **UI設計**：
  ```
  ┌─────────────────────────────────────┐
  │ 📍 起点レコードフィルター             │
  ├─────────────────────────────────────┤
  │ オブジェクト選択:                    │
  │  ( ) Dev_Service__c                 │
  │  ( ) Issue__c                       │
  │  (●) TechActionPlan__c              │
  │  ( ) FunctionRequirements__c        │
  │  ( ) Dev_Ticket__c                  │
  │                                     │
  │ レコード検索:                        │
  │  [🔍 検索...]                       │
  │   → レコードを検索して選択           │
  │                                     │
  │ 表示範囲:                            │
  │  ☑ 上流を表示                        │
  │  ☑ 下流を表示                        │
  │  ☐ 起点レコードのみ                  │
  │                                     │
  │ [✓ 適用] [× クリア]                 │
  └─────────────────────────────────────┘
  ```

* **実装パターン（5パターン）**：

  **パターン1: Dev_Service__c を起点**
  ```
  上流: なし（最上流のため）
  下流: Issue → TechActionPlan → FunctionRequirements → Dev_Ticket
  ```
  ユースケース：「このサービスに関連する全ての開発状況を確認」

  **パターン2: Issue__c を起点**
  ```
  上流: Dev_Service
  下流: TechActionPlan → FunctionRequirements → Dev_Ticket
  ```
  ユースケース：「この課題の背景と対応状況を確認」

  **パターン3: TechActionPlan__c を起点**
  ```
  上流: Issue → Dev_Service
  下流: FunctionRequirements → Dev_Ticket
  ```
  ユースケース：「この施策がなぜ必要で、どう実装されているか確認」

  **パターン4: FunctionRequirements__c を起点**
  ```
  上流: TechActionPlan → Issue → Dev_Service
  下流: Dev_Ticket
  ```
  ユースケース：「この機能要件の経緯と実装チケットを確認」

  **パターン5: Dev_Ticket__c を起点**
  ```
  上流: FunctionRequirements → TechActionPlan → Issue → Dev_Service
  下流: なし（最下流のため）
  ```
  ユースケース：「このチケットが発生した経緯を追跡」

* **表示範囲オプション**：
  * **上流を表示**：起点レコードより上位階層のレコードを表示
  * **下流を表示**：起点レコードより下位階層のレコードを表示
  * **起点レコードのみ**：起点レコード1つだけを表示（デバッグ用）
  * **上流・下流両方**（デフォルト）：全体の関連性を表示

* **実装上の考慮事項**：
  * 多対多の関係のため、再帰的にデータを取得
  * 無限ループ防止：訪問済みレコードをSetで管理
  * 最大深度制限：10階層まで（循環参照対策）
  * パフォーマンス：一括SOQLクエリでN+1問題を回避

### **5.4 フィルターの組み合わせ**

* **複数フィルターの同時適用**：
  * ステータスフィルター + オブジェクトタイプフィルター
  * ステータスフィルター + 起点レコードフィルター
  * 全てのフィルターを組み合わせ可能

* **適用順序**：
  1. 起点レコードフィルター（指定されている場合）
  2. オブジェクトタイプフィルター
  3. ステータスフィルター
  4. 孤立ノードの除外

* **フィルタークリア機能**：
  * 「クリア」ボタンで全フィルターをデフォルト状態に戻す
  * デフォルト状態：全データ表示（起点レコードなし、全ステータス、全オブジェクトタイプ）

## **6\. ユーザーインタラクション**

### **インタラクティブ性**

* **クリックで詳細表示**：ユーザーがノード（Dev_Service / Issue / TechActionPlan / FunctionRequirements / Dev_Ticket）をクリックすると、そのレコードの詳細情報が右側のサイドパネルに表示される。

  * 例：TechActionPlan__c をクリックすると、施策名、ステータス、担当者、期限、関連する課題（上流）と機能要件（下流）などを表示。

* **レコードページへの遷移**：
  * 詳細パネルのレコード名（Name）は**ハイパーリンク**として表示
  * クリックすると、Salesforce標準のレコード詳細ページに遷移
  * 新しいタブで開くか、同じタブで開くかは設定可能
  * URLフォーマット：`/lightning/r/{ObjectApiName}/{RecordId}/view`

* **フィルター外レコードの表示**：
  * 詳細パネルの関連レコード一覧には、**現在のフィルター条件で非表示になっているレコードも含めて表示**
  * フィルター外レコードの視覚的表現：
    - グレーアウト表示または淡色表示
    - アイコン表示（例：🚫 または ⊘）
    - ツールチップで「フィルター条件により非表示」と説明
  * これにより、グラフ上では見えない関連レコードの存在をユーザーに知らせる
  * ユースケース：
    - ステータス「done」をフィルター除外している状態で、Issue__c を選択
    - 詳細パネルには、完了済み（done）の TechActionPlan__c も表示される
    - ユーザーは「この課題に対して既に完了した施策がある」と認識できる

* **パスハイライト機能**：
  * 特定のノードを選択すると、そのノードに関連する**上流から下流まで全てのパス**を強調表示
  * クリティカルパス的に、開発の流れを視覚的に追跡できる
  * 例：ある Dev_Ticket__c を選択すると、そのチケットに至るまでの FunctionRequirements → TechActionPlan → Issue → Dev_Service の全てが強調表示される

* **ホバーで概要表示**：
  * ノードにマウスを重ねると、ツールチップで名前とステータスを簡易表示

* **エッジのインタラクション**：
  * エッジ（線）をクリックすると、その関係の詳細（作成日、関連する中間レコードの情報など）を表示

* **ズーム・パン操作**：
  * マウスホイールでズーム
  * ドラッグでパン
  * ノードのドラッグで位置調整可能

## **7\. データセキュリティとアクセス制御**

### **アクセス制限**

* **レコードベースのセキュリティ**：表示されるレコードは、ユーザーがアクセスできる権限を持つレコードだけに制限します。

  * **共有ルール**、**ロール階層**、**プロファイル設定**を使用して、アクセスできるデータを制限します。

  * Apex側でSOQLクエリを記述する際には、\*\*`with sharing`\*\*キーワードを使用して、Salesforceのセキュリティモデルに準拠します。

## **8\. 最小限の機能要件**

### **要件まとめ**

* **対象オブジェクト**：
  * 主要オブジェクト5種類：Dev_Service__c、Issue__c、TechActionPlan__c、FunctionRequirements__c、Dev_Ticket__c
  * 中間オブジェクト4種類：Relation_Support2Issue__c、Relation_Issue2Plan__c、Relation_Plan2Requirements__c、Relation_Requirements2Dev__c

* **階層構造**：5階層の開発プロセスフローを表現（ユーザサービス → 課題 → 施策 → 要件 → 開発チケット）

* **多対多の関係**：4つの中間オブジェクトを使って、各階層間の多対多の関係を実現。

* **ステータス管理**：各主要オブジェクトはステータス（todo/wip/done）を持ち、グラフ上で色分け表示。

* **データ取得**：
  * Apexを使って、5階層全体のデータを中間オブジェクトを介して取得
  * 効率的なSOQLクエリで、必要な全てのノードとエッジを取得

* **グラフ表示**：
  * LWCでノード（5種類の主要オブジェクト）とエッジ（4種類の関係）を視覚的に表示
  * Cytoscape.jsの階層型レイアウトで5階層構造を明確に表現
  * 形状でオブジェクトタイプ、色でステータスを識別

* **フィルター機能**：
  * **ステータスフィルター**（必須）：チェックボックスで複数選択可能
  * **オブジェクトタイプフィルター**：階層の表示/非表示を制御
  * **起点レコードフィルター**（重要）：
    - 5パターンの検索ロジック（各オブジェクトタイプごと）
    - 上流・下流の表示範囲を選択可能
    - 再帰的データ取得と無限ループ防止
  * フィルターの組み合わせ適用可能

* **インタラクション**：
  * ノードクリックで詳細情報表示
  * パスハイライト機能（選択ノードの上流・下流を強調）
  * ホバーでツールチップ表示
  * エッジクリックで関係の詳細表示
  * ズーム・パン・ドラッグ操作

* **詳細パネル機能**：
  * **レコード名のハイパーリンク**：クリックでSalesforce標準レコードページに遷移
  * **フィルター外レコードの表示**：
    - 現在のフィルター条件で非表示の関連レコードも詳細パネルに表示
    - グレーアウトやアイコンで視覚的に区別
    - ユーザーはグラフ上では見えない関連レコードの存在を認識可能
  * **全関連レコード一覧**：上流・下流の全レコードをハイパーリンク形式で表示

* **アクセス制限**：表示されるデータは、ユーザーの権限に基づいて制限される（`with sharing`使用）。

* **拡張性**：
  * 将来的に階層を追加できる設計
  * さらに複雑な分岐・合流パターンに対応可能
