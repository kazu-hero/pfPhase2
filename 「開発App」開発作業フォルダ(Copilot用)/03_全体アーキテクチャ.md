## **データフロー**

### **1. 初期表示時**

```
User → LWC (connectedCallback)
  → Apex: getFilteredGraph(no filters)
    → SOQL: 全オブジェクトのレコード取得
    → GraphDataBuilder: JSON整形
  ← Apex: { nodes: [], edges: [] }
← LWC: Cytoscape.jsでレンダリング
```

### **2. フィルター適用時**

```
User → フィルターUI操作
  → LWC: handleFilterChange()
    → Apex: getFilteredGraph(filterCriteria)
      → FilterHelper: フィルター適用
      → SOQL: 必要に応じて追加クエリ
    ← Apex: フィルター済みデータ
  ← LWC: グラフ再描画
```

### **3. ノードクリック時**

```
User → ノードクリック
  → LWC: handleNodeClick(nodeId)
    → Apex: getRelatedRecords(nodeId, objectType)
      → SOQL: 全関連レコード取得（フィルター無視）
      → FilterHelper: フィルター外判定
    ← Apex: { upstreamRecords: [], downstreamRecords: [] }
  ← LWC: 詳細パネル更新
```

## **セキュリティアーキテクチャ**

### **Apexレベル**
- **`with sharing`** を全てのコントローラで使用
- ユーザーのアクセス権限に基づいたレコード表示
- CRUD/FLS（Field Level Security）チェック

### **LWCレベル**
- ユーザーがアクセスできないレコードは非表示
- エラーメッセージでセンシティブ情報を漏らさない

### **データ取得クエリ**
```apex
// 正しい例（セキュリティ遵守）
public with sharing class RelationGraphController {
    @AuraEnabled(cacheable=true)
    public static GraphData getFilteredGraph(FilterCriteria criteria) {
        // ユーザーの権限に基づいて自動的にフィルタリング
        List<Issue__c> issues = [SELECT Id, Name FROM Issue__c];
        // ...
    }
}
```

## **パフォーマンス最適化戦略**

### **1. SOQL最適化**

**NG例（N+1問題）:**
```apex
for (Issue__c issue : issues) {
    List<Relation_Issue2Plan__c> relations = [
        SELECT TechActionPlan__c FROM Relation_Issue2Plan__c 
        WHERE Issue__c = :issue.Id
    ];
}
```

**OK例（一括取得）:**
```apex
Set<Id> issueIds = new Set<Id>();
for (Issue__c issue : issues) {
    issueIds.add(issue.Id);
}
List<Relation_Issue2Plan__c> relations = [
    SELECT Issue__c, TechActionPlan__c, TechActionPlan__r.Name
    FROM Relation_Issue2Plan__c 
    WHERE Issue__c IN :issueIds
];
```

### **2. データ量制限**

```apex
public static final Integer MAX_NODES = 500;
public static final Integer MAX_DEPTH = 10;

// 再帰処理での深度チェック
private static void fetchRecursive(Id recordId, Integer depth) {
    if (depth > MAX_DEPTH) {
        return; // 深度制限
    }
    // ...
}
```

### **3. キャッシング**

```apex

    // 同じ条件での再クエリを防ぐ
```

### **4. フロントエンド最適化**

- 大量ノード時の仮想レンダリング検討

## **Governor Limits対策**

### **主要な制限と対策**

| 制限項目 | 上限値 | 対策 |
|---------|--------|------|
| SOQL クエリ数 | 100 | 一括クエリ、SOQLの統合 |
| レコード取得数 | 50,000 | ページネーション、フィルター必須化 |
| ヒープサイズ | 6MB | データ整形時の最適化、ストリーム処理 |
| CPU時間 | 10,000ms | 複雑な計算の簡略化、非同期処理検討 |

### **実装上の注意**

```apex
// SOQL数カウント
System.debug('SOQL Queries: ' + Limits.getQueries());
```

## **エラーハンドリング戦略**

### **Apex側**

```apex
@AuraEnabled
public static GraphData getFilteredGraph(FilterCriteria criteria) {
    try {
        // ビジネスロジック
        return buildGraphData(criteria);
    } catch (QueryException e) {
        throw new AuraHandledException('データ取得エラー: ' + e.getMessage());
    } catch (Exception e) {
        System.debug(LoggingLevel.ERROR, 'Unexpected error: ' + e.getStackTraceString());
        throw new AuraHandledException('予期しないエラーが発生しました');
    }
}
```

### **LWC側**

```javascript
async loadGraphData() {
    try {
        const data = await getFilteredGraph({ criteria: this.filterCriteria });
        this.renderGraph(data);
    } catch (error) {
        this.showToast('エラー', error.body.message, 'error');
        console.error('Graph load error:', error);
    }
}
```

## **📌 アーキテクチャ設計の原則**

1. **疎結合** - コンポーネント間の依存を最小化
2. **単一責任** - 各クラス/コンポーネントは1つの責務のみ
3. **再利用性** - 汎用的な処理は共通化
4. **テスタビリティ** - テストしやすい設計
5. **スケーラビリティ** - データ増加に耐えられる設計

**この設計に従うことで、保守性と拡張性の高いシステムを実現します。**
